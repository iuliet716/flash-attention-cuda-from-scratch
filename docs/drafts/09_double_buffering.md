# Step 9. Double Buffering — cp.async (한글 초안)

> 최종 영어 문서 작성용 참고 초안. 이 문서의 핵심은 기법 자체보다
> "1차 시도 실패(0.61x) → 원인 진단 → 재설계(1.64x)" 서사다.

## 1. 개념

### 문제: 로드와 계산의 교대

step 08의 타일 루프는 [K/V 로드 → 배리어 → 계산 → 배리어]의 반복이다.
로드하는 동안 텐서코어가 놀고, 계산하는 동안 메모리 경로가 논다.
둘을 겹치면(overlap) 그만큼 공짜 시간이 나온다.

### cp.async — 백그라운드 복사

Ampere(sm_80)부터 생긴 비동기 복사 명령. 일반 로드는
"HBM → 레지스터 → smem 저장"으로 스레드를 붙잡지만, `cp.async`는
**HBM → smem 직행**을 백그라운드로 걸어두고 스레드는 다음 일을 계속한다.

프로그래밍 모델 (`<cuda_pipeline.h>`):

- `__pipeline_memcpy_async(dst_smem, src_global, 16)` — 16B 복사 예약
- `__pipeline_commit()` — 지금까지 예약분을 그룹 하나로 마감
- `__pipeline_wait_prior(N)` — "최신 N개 그룹만 빼고" 완료될 때까지 대기

### 더블 버퍼링 파이프라인

버퍼를 2개 두고, 타일 t를 계산하는 동안 타일 t+1을 다른 버퍼로 프리페치:

```
버퍼0:  [load T0]      [compute T0]              [load T2]  [compute T2]
버퍼1:            [load T1]        [compute T1]            [load T3] ...
                  ^^^^^^^ compute T0와 겹침
```

### 1차 시도 실패: 0.61x — 이 문서의 하이라이트

step 08 구조(BR=16, 4워프, smem ~55KB)에 그대로 더블 버퍼를 얹었더니
38 → 62ms로 **오히려 느려졌다**. 원인:

- smem이 ~93KB로 2배 → SM당 상주 블록 2~3개 → **1개**
- GPU는 원래 SM에 블록 여러 개를 얹어 "한 블록이 로드하는 동안 다른 블록이
  계산"하는 **블록 간 오버랩을 공짜로** 제공하고 있었다
- 블록 내 오버랩(더블 버퍼)을 사는 대가로 그 공짜를 잃었다 — 잃은 게 더 컸다

교훈: **명시적 더블 버퍼링은 "한 블록이 SM을 소유하는" 설계에서만 이득이다.**
같은 기법이 점유율 문맥에 따라 0.61x도 1.64x도 된다.

### 재설계: 블록이 SM을 소유하게

- BR 16→**64**, 워프 4→**8**: 처음부터 1블록/SM 설계. 이제 잃을 블록 간
  오버랩이 없으니 cp.async가 순수한 이득이 된다
- 덤: BR 4배 → K/V를 Q블록당 한 번씩 재읽으므로 **HBM 재읽기 트래픽 1/4**
- BC 64→**32**: d=128에서 더블 버퍼 포함 smem이 99KB(블록당 한도) 안에
  들어가기 위한 조정

워프 분업도 전면 재작성이 필요했다 (BR/BC/WARPS는 독립 조절 가능한 상수가
아니라 세 페이즈의 인덱스 산술에 묶여 있다):

- QK^T: S(64×32)를 4×2 그리드로, **워프당 16×16 타일 정확히 1개**
- softmax: 워프당 8행씩 8워프가 분담 (기존: 워프 0 혼자)
- PV: 워프 (r, c)가 O의 [16r, 16r+16)행 × [c·d/2, +d/2)열 담당
- 이 관계들을 `static_assert`로 박아, 상수만 바꾸면 컴파일이 실패하게 했다

(여담 소재: 처음에 상수만 BR=64/8워프로 바꿨다가 softmax가 32행까지만 처리되어
`l=0 → 0/0 = NaN`이 났던 실패담 — "상수가 독립 파라미터가 아니다"의 실증)

### 결과

41.6 → 25.3ms (1.64x). fused 커널이 처음으로 최강 unfused 구현(step 02,
~29ms)을 추월한 지점이기도 하다.

## 2. 코드 구현 설명

### 비동기 로더

```cuda
auto load_kv_async = [&](int stage, int tile) {
    for (각 스레드가 맡은 16B 청크) {
        if (행 < N)  __pipeline_memcpy_async(smem 위치, 글로벌 위치, 16);
        else         smem 위치 = 0;   // 경계 밖은 동기 store로 zero-fill
    }
};
```

zero-fill을 일반 store로 하는 이유: cp.async는 "복사"만 할 수 있어서
패딩은 직접 써야 한다. 같은 스레드의 쓰기이므로 뒤의 배리어로 정렬된다.

### 파이프라인 골격

```cuda
load_kv_async(0, 0);            // 프롤로그: 첫 타일 프리페치
__pipeline_commit();

for (int t = 0; t < n_tiles; ++t) {
    if (t + 1 < n_tiles) {
        load_kv_async((t+1) % 2, tile + BC);   // 다음 타일 발행
        __pipeline_commit();
        __pipeline_wait_prior(1);   // "타일 t까지" 완료 대기 (t+1은 진행 중)
    } else {
        __pipeline_wait_prior(0);   // 마지막: 전부 대기
    }
    __syncthreads();                // (1)
    ... 버퍼 t%2로 QK^T, softmax, PV ...
    __syncthreads();                // (2)
}
```

### 배리어 두 개가 각각 지키는 것

- **(1) wait 직후**: `wait_prior`는 "그 스레드가 발행한" 복사만 보장한다.
  다른 스레드가 채운 smem을 읽으려면 블록 전체 배리어가 필요하다.
- **(2) 루프 끝**: 다음 반복의 프리페치가 이 버퍼를 덮어쓰기 전에,
  모든 워프가 이번 소비를 끝냈음을 보장한다.

버퍼 안전성 논증: 반복 t에서 덮어쓰는 버퍼는 (t+1)%2 — 이 버퍼를 마지막으로
읽은 것은 반복 t-1이고, t-1은 배리어 (2)로 끝났으므로 겹침이 없다.

### smem 예산 (재설계의 숫자 근거)

- 실패 버전: BR=16/BC=64 + 2스테이지 → d=128에서 ~93KB → 1블록/SM
- 재설계: BR=64/BC=32 + 2스테이지 → d=64: ~60KB, d=128: ~98.8KB
  (sm_120 블록당 한도 99KB에 턱걸이; 48KB 초과라 `cudaFuncSetAttribute` 필요)

## 3. 벤치마크 참고

- **필수**: 메인 표 (08→09, ~1.6x) + **실패 버전의 수치(62.6ms, 0.61x)**.
  이미 측정해 둔 기록이 있다. 성공 수치만 실으면 평범한 문서지만,
  실패 수치가 나란히 있어야 "점유율" 논증이 완성된다.
- **선택**: ncu에서 08 vs 09의 `long_scoreboard` stall(글로벌 메모리 대기)
  감소와 achieved occupancy 비교.
