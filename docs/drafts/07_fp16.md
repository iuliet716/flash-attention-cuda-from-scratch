# Step 7. Half-Precision — FP16 (한글 초안)

> 최종 영어 문서 작성용 참고 초안.

## 1. 개념

### 왜 fp16인가

FP16(half)은 1비트 부호 + 5비트 지수 + 10비트 가수, 총 2바이트다.
유효숫자 약 3자리, 최대값 약 65504.

원소 하나가 절반 크기가 되면:

- HBM 트래픽 절반
- shared memory 사용량 절반 (같은 SRAM에 2배 큰 타일 가능)
- 16B 청크(float4) 하나에 4개가 아니라 **8개** 원소

즉 fp16은 "정밀도를 지불하고 대역폭과 용량을 사는" 거래다.

### 전략: 저장만 fp16, 계산은 fp32

fp16으로 수천 개 항을 그대로 누적하면 오차가 쌓이고, softmax의 exp는 정밀도에
민감하다. 그래서 경계를 명확히 긋는다:

- **메모리에 있는 동안만 fp16**: Q, K, V, O의 HBM 저장과 smem 타일
- **레지스터에 올라오면 즉시 fp32로 변환**: 내적, exp, 누적 전부 fp32
- softmax 상태(m, l)와 PV 누산기는 **항상 fp32 레지스터**

이렇게 하면 반올림 오차가 "각 원소를 fp16으로 한 번 표현하는" 수준에 머물고
N이 커져도 누적되지 않아, 정확성 검사(rtol/atol 1e-2)를 통과한다.
출력도 half로 반환한다 — PyTorch SDPA FlashAttention과 같은 관례다.

### 결과가 1.02x뿐인 이유, 그런데도 중요한 이유

트래픽을 절반으로 줄였는데 시간은 거의 그대로다(64.2→62.9ms). 이유는 단순하다.
**이 시점의 커널은 대역폭 병목이 아니다** — 연산과 지연시간이 병목이라, 절반이 된
트래픽은 임계 경로에 거의 반영되지 않는다.

이 리포를 관통하는 교훈의 한 사례다: **기법의 효과는 그 기법이 줄이는 자원이
현재 병목일 때만 시간으로 나타난다.**

그럼에도 이 단계가 필수인 이유: 텐서코어(step 08)는 **fp16 입력을 요구**한다.
step 07의 진짜 역할은 당장의 속도가 아니라 다음 단계의 문을 여는 것이다.
(step 03 online softmax가 당장은 0.98x였지만 fusion의 전제였던 것과 같은 패턴)

참고: 벤치마크 시간에는 forward() 내부의 fp32→fp16 변환 비용도 포함된다.

## 2. 코드 구현 설명

### 타일과 청크 단위

```cuda
extern __shared__ __align__(16) __half smem[];
const int d8 = d / 8;   // 16B 청크 = 8 halves
```

글로벌↔smem 이동은 여전히 float4(16B)로 하되, 그 안에 half 8개가 들어 있다.
스위즐도 8-half 청크 단위가 되고, "청크 수가 8의 배수" 조건이
`d/8 % 8 == 0` → **`d % 64 == 0`** 제약으로 강화된다.

### 내적: fp16을 fp32로 풀어서 누적

```cuda
const float4 a = q8[c];              // half 8개가 든 16B
const __half2* ah = reinterpret_cast<const __half2*>(&a);
for (int j = 0; j < 4; ++j) {
    const float2 qf = __half22float2(ah[j]);  // half 2개 → float 2개
    const float2 kf = __half22float2(bh[j]);
    dot += qf.x * kf.x + qf.y * kf.y;         // 누적은 fp32
}
```

`__half22float2`가 half 2개를 float 2개로 변환하는 내장 함수다.
로드는 16B 한 번, 산술은 fp32.

### PV와 출력

```cuda
acc[i] += pc * __half2float(Vtile[c * d + k]);      // 읽을 때 변환
...
Ob[...] = __float2half(acc[i] * inv_l);             // 쓸 때 변환
```

### 호스트 코드

```cpp
auto q_contig = q.to(torch::kHalf).contiguous();
...
launch_fused_attention(
    reinterpret_cast<const __half*>(q_contig.data_ptr<at::Half>()), ...);
```

torch의 `at::Half`와 CUDA의 `__half`는 메모리 표현이 같아 reinterpret으로 넘긴다.
출력 텐서도 half 옵션으로 만들어 그대로 반환.
